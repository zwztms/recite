# 前端缓存类 Bug — 诊断指南

> 日期：2026-06-20 | 事件：模块选择器空白 + 背诵调用失败

---

## 一、错误现象

| 现象 | 表现 |
|------|------|
| 模块选择器空白 | ModeSelector 不显示任何模块 |
| 点击背诵报错 | `Uncaught (in promise) Error: 服务器内部错误` at `index.js:31` |
| Console 无报错 | 模块列表 API 调用失败但被 try-catch 吞掉 |

---

## 二、实际根因

**Vite HMR 缓存 + 浏览器缓存**导致前端运行的是旧版本打包代码。后端 API 全部正常（curl 验证 4/4 通过），成就系统修改与背诵/知识库模块零交集。

**证据链**：
1. `curl` 测试 `/admin/knowledge/modules` → 19 modules ✅
2. `curl` 测试 `/recite/start` (CATEGORY/RANDOM/REVIEW) → 全部 code=0 ✅
3. 成就改动文件与 recite/knowledge 代码无任何 import 关系 ✅
4. 重启前端 Vite (`npx vite build` → `npx vite --host`) 后问题消失 ✅

| 阶段 | AI 行为 | 问题 |
|:--:|------|------|
| 1 | 用户说"模块列表失败，与成就修改有关" | AI 立即进入"自查代码改动"模式 |

## 四、为什么会绕远路

### 4.1 AI 的认知偏差

1. **确认偏误**：用户说"与成就修改有关"，AI 就在成就代码里找问题，忽略了"巧合"的可能性
2. **过度分析**：代码逻辑正确但还在反复 read/grep，没有先验证最简单的假设（缓存）
3. **信任 curl 但不信任"无改动"**：curl 证明了后端正常、代码未改，却还在怀疑隐藏的依赖关系

### 4.2 这类 Bug 的特征

前端空白/报错但后端 curl 正常 → **100% 前端问题**。具体子类：

| 子类 | 特征 | 最简验证 |
|------|------|------|
| **构建缓存** | 改动代码后页面不更新 | 重启 Vite / `rm -rf dist node_modules/.vite` |
| **浏览器缓存** | 同一 URL 别人正常 | Ctrl+Shift+R 硬刷新 |
| **HMR 状态污染** | 多次热更新后状态异常 | 重启 Vite |
| **localStorage 过期** | token 存在但请求 401/500 | Application → 检查 token 值 |

## 五、与 AI 交流这类 Bug 的建议

### 5.1 开局模板

```
现象：{具体描述}
浏览器 F12 Console 报错：{粘贴完整错误堆栈}
Network 标签 {具体请求} 的 Response：{粘贴}
我怀疑与 {X 修改} 有关，但不确定。
```

### 5.2 关键信息四要素

| 信息 | 来源 | 为什么重要 |
|------|------|------|
| **Console 报错** | F12 → Console | 定位 JS 错误的精确文件和行号 |
| **Network 请求** | F12 → Network → 点具体请求 → Response | 区分"后端返回错误"还是"前端没发请求" |
| **是否硬刷新过** | Ctrl+Shift+R | 排除浏览器缓存 |
| **后端 API 独立测试** | curl 或 Postman | 区分前后端责任 |

### 5.3 当你怀疑"与某修改有关"时

```
先要求 AI 做影响面分析：
"列出 {X 修改} 改动的所有文件，逐个说明是否与 {Y 功能} 有 import/调用关系。
然后用 curl 独立验证 Y 功能的后端 API。"
```

这样可以快速排除或确认关联。

### 5.4 当 AI 反复读代码但无进展时

直接打断：
```
"后端 curl 已经通过了，现在聚焦前端。先尝试重启 Vite + 硬刷新浏览器，
如果还不行，再查前端代码。"
```

## 六、结论

| 维度 | 事实 |
|------|------|
| 根因 | Vite/浏览器缓存，前端运行旧版构建 |
| 与成就系统关系 | **无关**——0 个共同文件，0 条 import 链 |
| 解决方式 | 重启前端 Vite |
| 可避免的消耗 | ~20 轮对话、多次代码审查、多次 curl 测试 |

> 核心教训：**curl 验证后端 → 重启前端 → 再查代码**。先排除最简单的，再深挖复杂的。
